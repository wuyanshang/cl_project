# 资产盘点 - 数据基线建设方案

## 一、项目背景

### 1.1 业务目标

将字段信息（系统、表、字段的中英文名）匹配挂载到对应的资产目录（主题域 → 二级分类 → 三级分类 → 数据资产）。

### 1.2 当前架构

```
输入(系统/表/字段 中英文名)
        │
        ▼
   五路召回(ES)
   ① 字段中文名 → asset_embedding KNN + 语义精排
   ② 字段中文名 → asset match_phrase
   ③ 字段中文名+表中文名 → category_embedding KNN + 语义精排
   ④ 字段中文名+表中文名 → asset_embedding KNN + 语义精排
   ⑤ 字段中文名 → category BM25
        │
        ▼
   AI节点从候选列表中选择匹配资产
```

### 1.3 当前性能基线

| 类别 | 数量 | 占比 |
|---|---|---|
| 召回且TOP1命中 | - | -% |
| 召回且TOP2-3命中 | - | -% |
| 召回但TOP1~3均未命中 | - | -% |
| 无法召回 | - | -% |
| **合计** | **-** | **100%** |

> 请根据实际数据集填写上表。

---

## 二、数据基线建设目标

**核心目标**：在数据进入召回流程前，识别并处理质量不达标的数据，分为：

- **清洗**：数据可修复（格式异常），清洗后重新进入流程
- **拦截**：数据质量过低，无法支撑准确匹配，标记并跳过
- **降级**：数据有歧义但可尝试，标记低置信度后继续流程

---

## 三、执行计划

### 第1天上午：数据清洗 + AI首轮标注

#### 任务3.1.1 代码层 - 正则清洗逻辑

对字段中文名、表中文名做预处理，在进入召回/AI标注之前执行。

**清洗规则（按顺序执行）：**

| 序号 | 规则 | 正则/逻辑 | 示例 |
|---|---|---|---|
| C1 | 去除换行符后的描述内容 | 取第一个 `\n` 前的文本 | `字段名\n1.值A 2.值B` → `字段名` |
| C2 | 去除枚举/码表描述 | 去除 `1.`/`2.`/`枚举值`/`码表` 及之后内容 | `状态\n1.有效 2.无效` → `状态` |
| C3 | 提取中英混合名的中文部分 | 去除连续英文+空格前缀，保留中文 | `English Prefix中文名称` → `中文名称` |
| C4 | 去除括号内补充说明 | 去除 `（...）` 中的补充内容（保留核心名称） | `业务日期（生效日）` → `业务日期` |
| C5 | 去除多值并列中的次要项 | 取第一个 `、` 或 `/` 前的内容 | `国籍/民族` → `国籍` |
| C6 | 表中文名清洗 | 去除 `\n`、`（）` 内的英文说明 | `TABCODE(表中文名)\nEnglish Desc` → `表中文名` |

> 以上清洗规则为通用模板，请根据实际数据集的字段命名模式进行调整和补充。

**注意**：清洗后保留原始值，清洗后的值作为新列用于后续流程，原始值用于审计追溯。

#### 任务3.1.2 AI层 - 全量歧义标注

用下方【提示词A - 歧义检测】对清洗后的全量数据做标注。

**执行方式**：
- 每批50-100条（避免上下文过长影响质量）
- 输入：清洗后的字段中文名、原始字段英文名、清洗后的表中文名、原始表英文名、系统名
- 输出：每条数据的歧义类型、置信度、理由

### 第1天下午：交叉分析 + AI归纳规则

#### 任务3.2.1 交叉分析

将AI标注结果与已有的召回结果做交叉，生成四象限矩阵：

```
                    AI判定=PASS       AI判定=有歧义
TOP4命中           ┌─────────────┬──────────────┐
                   │  正常放过     │  疑似误拦      │
                   │  (统计数量)   │  (需人工审核)   │
                   ├─────────────┼──────────────┤
TOP4未命中/        │  疑似漏拦     │  正常拦截       │
无法召回           │  (需人工审核)  │  (统计数量)     │
                   └─────────────┴──────────────┘
```

**输出交付物**：
- 四象限数量统计
- 疑似误拦样本列表
- 疑似漏拦样本列表

#### 任务3.2.2 AI归纳规则（从好数据学规则）

用下方【提示词B - 规则归纳】，将好数据和坏数据一起喂给AI，让AI自动归纳区分规则。

### 第2天上午：规则整合 + 人工审核

#### 任务3.3.1 规则整合

合并三个来源的规则：
1. 人工预设的 A0~A3b 规则
2. AI从好/坏数据对比中归纳出的新规则
3. AI从矛盾样本分析中发现的修正建议

**整合原则**：
- 去除重复规则
- 优先保留高区分度规则（B组违反率高、A组违反率低）
- 人工确认每条规则的合理性

#### 任务3.3.2 人工审核

重点审核：
1. **AI归纳的新规则**：是否有业务意义、是否可执行
2. **疑似误拦样本**：确认是否真的不该拦（可能需要放宽规则）
3. **疑似漏拦样本**：确认是否真的应该拦（可能需要新增规则）

### 第2天下午：第二轮标注 + 统计

#### 任务3.4.1 第二轮AI标注

用整合后的规则重新标注全量数据。

#### 任务3.4.2 统计与填表

输出每条规则的拦截统计：

| 规则 | 总命中数 | 基线拦截个数 | 占比 | 与TOP4的交叉验证 |
|---|---|---|---|---|
| A0-格式异常 | ? | ? | ?% | 误拦率?% |
| A1-中英文矛盾 | ? | ? | ?% | 误拦率?% |
| ... | | | | |

### 第3天：确定阈值 + 输出交付物

#### 任务3.5.1 拦截策略定级

| 歧义类型 | 处置策略 | 理由 |
|---|---|---|
| A0（格式异常） | **清洗后放行** | 清洗可解决，不需要拦截 |
| A1（中英文矛盾）- HIGH置信度 | **拦截** | 数据本身有错，匹配必错 |
| A1（中英文矛盾）- LOW置信度 | **降级放行** | 可能是AI误判 |
| A2（缩写不明确） | **降级放行** | 不影响中文名召回 |
| A3a（粒度不足）且表名无法消歧 | **拦截** | 无法确定目标资产 |
| A3a（粒度不足）且表名可消歧 | **放行**（联合表名查询） | 上下文可补充 |
| A3b（描述不完整） | **放行** | 不直接影响召回 |

#### 任务3.5.2 最终交付物

1. 基线拦截统计表（填入Excel的"基线拦截个数"和"占比"列）
2. 清洗规则文档（正则规则）
3. 歧义检测规则文档（提示词）
4. 规则迭代记录（每轮规则变更的原因和效果）

## 四、提示词

### 提示词A - 歧义检测

```
## 角色

你是数据质量检测专家，负责评估字段元数据的质量，判断其是否存在歧义。
你的判断结果将用于决定该字段数据是否应该进入资产匹配流程。

## 输入数据格式

每条数据包含以下字段：
- system_name: 系统名称
- table_en: 源表英文名
- table_cn: 源表中文名（已清洗）
- field_en: 源字段英文名
- field_cn: 源字段中文名（已清洗）
- field_cn_raw: 源字段中文名（原始值，清洗前）

## 检测的歧义类型

### A0 - 数据格式异常（优先检测）

字段中文名（原始值）的内容格式异常，不是一个规范的字段命名。

**判定标准：**

1. **中文名含描述性内容**：原始字段中文名中包含换行符、枚举值定义、码表说明等非名称内容
   - 例: field_cn_raw=`字段名\n1.值A\n2.值B\n码表：XXX` → 实际字段名应为"字段名"
   - 例: field_cn_raw=`业务类型\n枚举值：类型A、类型B、类型C`

2. **中文名为中英混合且语义不连贯**：原始字段中文名中混入了英文系统术语
   - 例: field_cn_raw=`English Prefix中文名称` → 实际中文含义是"中文名称"
   - 例: field_cn_raw=`System Field Label业务含义`

3. **中文名列填写的是纯英文**：没有中文释义
   - 例: field_cn_raw=`colname`（中文名列却是英文缩写）

**输出要求：** 识别为A0时，请确认清洗后的field_cn是否合理，如不合理请给出你的建议。

### A1 - 中英文名不匹配

字段的中文名和英文名描述的不是同一个业务概念。

**判定标准：**

1. **直接矛盾**：英文名和中文名指向完全不同的业务概念
   - 例: field_en=`planName`, field_cn=`保险合同全称` → planName 指"险种名称"，不是合同全称

**注意：** 英文名是系统内部缩写编码（如ORGID、CRTDTE、STATUS）时，不要仅因为缩写不直观就报A1。只有当英文名有明确的业务语义且与中文名矛盾时才报。

### A2 - 缩写不明确

字段名包含缩写，该缩写有多种可能的展开含义，且上下文不足以消歧。

**判定标准：**

1. **多义缩写**：缩写在保险/金融/IT领域有多个常见含义，且系统名+表名+中文名都不足以消歧
   - 例: 表名`wp` → 全险? 寿险? 豁免? 工作计划?
   - 例: `tp` → 第三者? 交易价格? 转移价格?

### A3a - 字段名粒度不足

字段中文名过于笼统，在不同业务上下文中可能对应多个不同的数据资产，仅靠字段名+表名无法确定具体指向哪个资产。

**判定标准：**

1. **泛化命名**：字段名是通用词，缺少业务实体限定
   - 例: field_cn=`状态`，table_cn=`某业务信息表` → 哪种状态？多种业务实体都有状态
   - 例: field_cn=`编号`，table_cn=`某管理表` → 哪个实体的编号？

2. **编码/名称不分**：只说了业务概念，不清楚存储的是编码还是名称
   - 例: field_cn=`国籍`, field_en=`nation` → 国籍编码？国籍名称？
   - 例: field_cn=`分公司`, field_en=`BRANCH` → 分公司编码？名称？简称？

**重要：** 此规则必须在尝试"表名消歧"和"英文名消歧"后仍无法确定时才报。

### A3b - 字段描述不完整

字段名本身能理解，但缺少必要的补充信息。

**判定标准：**

1. **枚举值含义不透明**：字段是枚举/状态类型，但不知道各值的含义
   - 例: field_cn=`状态`, field_en=`STATUS`
2. **业务主体不明确**：不确定描述的是哪个业务实体的属性
   - 例: field_cn=`性别` → 投保人?被保人?代理人?

**注意：** 如果同时触发A3a和A3b，只报A3a。

## 通用排除规则（以下情况不算歧义）

### 上下文消歧
- **表名可推导明确含义**：能通过表名确定字段含义时，不报A3a/A3b
  - 例: table_cn=`投保人信息表` + field_cn=`性别` → 明确是投保人性别
  - 例: table_cn=`索赔前调查` + field_cn=`状态` → 明确是调查状态
  - ⚠️ 限制：表名本身必须是可理解的中文。如果表名是纯英文编码且无中文说明，不认为可消歧

- **字段中英文名互补消歧**：合起来可明确含义时，不报A3a/A3b
  - 例: field_en=`nationality_code` + field_cn=`国籍` → 明确是编码
  - 例: field_en=`HSPCOD` + field_cn=`就诊医院` → 明确是编码

- **备注/预留字段**：英文名符合备注(remark/filler/memo)或预留字段命名时，不检查A1

- **中文名含英文但含义明确**：整体含义可理解时，不单独报A2
  - 例: field_cn=`VIP客户标志` → 虽含英文但整体含义明确

## 输出格式

对每条数据输出JSON：

{
  "index": 数据序号,
  "ambiguity_types": ["A0", "A3a"],  // 歧义类型列表，无歧义则为["PASS"]
  "confidence": "HIGH",               // HIGH/MEDIUM/LOW
  "clean_suggestion": "分公司",        // 仅A0时填写，清洗建议
  "reason": "一句话判定理由",
  "should_intercept": true             // 综合判断：是否建议拦截
}

## 拦截建议规则

- A0 → should_intercept=false（清洗可解决）
- A1 HIGH置信度 → should_intercept=true
- A1 MEDIUM/LOW → should_intercept=false
- A2 → should_intercept=false
- A3a 且表名无法消歧 → should_intercept=true
- A3a 且表名可消歧 → should_intercept=false
- A3b → should_intercept=false
- PASS → should_intercept=false

## 注意事项

1. **先检测A0**。如果有格式异常，先确认清洗是否合理，再基于清洗后的field_cn判断其他歧义。
2. **结合上下文判断**。不要孤立看字段名，结合系统名、表名一起判断。
3. **中文名能消歧就不报**。即使英文名是缩写。
4. **表名能限定就不报**。但表名本身必须可理解。
5. **只报影响匹配的歧义**。不是所有不完美都需要报告。
6. **批量处理保持一致性**。同一类问题判定标准一致。
```

### 提示词B - 规则归纳（从好数据vs坏数据学习）

```
## 角色

你是数据质量规则工程师。你需要通过对比"高质量数据"和"低质量数据"，归纳出可执行的数据质量检测规则。

## 任务

以下有两组数据：

【A组 - 高质量数据】
这些数据在资产匹配流程中成功匹配到了正确的数据资产（TOP1命中）。

【B组 - 低质量数据】
这些数据在资产匹配流程中无法召回任何候选资产，匹配完全失败。

## 分析要求

### 第一步：特征分析

分别分析A组和B组在以下维度上的特征分布：

1. **字段中文名长度**：A组和B组的长度分布是否有显著差异
2. **字段中文名构成**：纯中文/中英混合/含特殊符号的比例差异
3. **字段中英文名关联度**：英文名是否可被解读为中文名的翻译/缩写
4. **字段名语义清晰度**：是否为泛化词（如"状态""编号""类型"）
5. **表名信息量**：表中文名是否提供了足够的业务上下文
6. **字段名与常见资产命名模式的匹配度**：是否符合"业务概念+属性类型"的命名规范（如"离职日期""国籍代码"）

### 第二步：差异归纳

找出A组和B组之间最显著的差异点，即：
- A组普遍满足但B组普遍不满足的特征
- B组特有的但A组几乎没有的特征

### 第三步：规则生成

基于差异分析，生成可执行的数据质量检测规则。

## 输出格式

对每条规则输出：

| 字段 | 内容 |
|---|---|
| 规则编号 | R1, R2, R3... |
| 规则描述 | 用自然语言描述规则内容 |
| 检测方法 | 正则/关键词匹配/长度阈值/AI判断 |
| A组违反率 | 该规则在高质量数据中的命中率（应该低，<10%为佳） |
| B组违反率 | 该规则在低质量数据中的命中率（应该高，>30%为佳） |
| 区分度 | B组违反率 - A组违反率（越高越好） |
| 建议处置 | 拦截 / 清洗 / 降级 / 仅标记 |

**规则质量要求：**
- 生成5-10条规则
- 每条规则的区分度（B违反率 - A违反率）应 > 20%
- 规则之间应互补，不要高度重叠
- 规则应该是可编程执行的（正则、长度判断等），不要过于主观

## 注意事项

1. 不要被个别异常样本影响，关注整体规律
2. 如果某个特征在两组中分布差异不大，不要强行生成规则
3. 优先输出区分度最高的规则
```

### 提示词D - 矛盾样本分析（漏拦分析）

```
## 角色

你是数据质量规则优化专家。你需要分析通过了歧义检测但实际匹配失败的数据，找出规则需要补充的地方。

## 背景

以下数据被歧义规则判定为PASS（质量合格），但实际在匹配流程中无法召回任何候选或匹配完全错误。

这说明可能存在规则未覆盖的数据质量问题。

## 输入数据

每条数据包含：
- 字段信息：system_name, table_en, table_cn, field_en, field_cn, field_cn_raw
- AI标注结果：ambiguity_types=["PASS"], confidence, reason
```

## 五、迭代机制

### 5.1 单轮迭代流程

```
规则集(V_n) → AI标注全量数据 → 交叉分析
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
               正常放过          疑似误拦          疑似漏拦
              (不处理)      (提示词C分析)     (提示词D分析)
                                    │               │
                                    ▼               ▼
                              修正建议           新增建议
                                    │               │
                                    └───────┬───────┘
                                            ▼
                                    人工审核确认
                                            │
                                            ▼
                                    规则集(V_n+1)
```

### 5.2 迭代终止条件

满足以下**任一**条件时停止迭代：

1. 误拦率 < 5%（被拦截的数据中，TOP4能成功匹配的 < 5%）
2. 两轮迭代之间规则无实质变化
3. 已完成3轮迭代（时间约束）

## 六、后续优化方向（基线完成后）

| 优化项 | 说明 | 预期收益 |
|---|---|---|
| Query扩展 | 用LLM对清洗后的字段名做同义词扩展再召回 | 提升召回率 |
| RRF融合排序 | 用Reciprocal Rank Fusion替代简单拼接去重 | 提升TOP1命中率 |

| 资产侧同义词扩展 | 对资产名生成别名并建立embedding | 提升召回率 |
